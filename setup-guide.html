2

Create a Google Cloud project

Why do I need Google Cloud for this?

Google Ads doesn't let apps connect directly. Instead, Google requires every app that reads Ads data to go through Google Cloud Platform (GCP) — their developer infrastructure. Think of it as the gatehouse between your AI agent and the Ads data.

Here's what each piece does and why it's needed:

What you create What it does Analogy
GCP Project A container that holds all your API credentials and settings. One project per client or per setup. Like a folder on your computer — everything for this setup lives inside it
Google Ads API (enabled) Tells Google "this project is allowed to talk to Google Ads." Without this, API calls are rejected. Flipping the switch that unlocks the Ads data door
OAuth consent screen (Phase 3) The login screen Google shows when you authenticate. It says "this app wants to read your Ads data — allow?" The "Sign in with Google" popup you see on other apps
OAuth credentials (Phase 4) A Client ID + Secret pair that identifies your app to Google. Downloaded as a JSON file. Like a username and password — but for your app, not for you
Developer Token (Phase 5) A key from Google Ads itself (not GCP) that grants API-level access to Ads data. Requires an MCC account. The VIP pass that says "Google approves this app to read Ads data"

All of this is free. You don't need billing enabled, a credit card, or a paid GCP plan. The free tier covers everything needed here.

Why Google's official MCP server?

This guide uses Google's own open-source MCP server (github.com/googleads/google-ads-mcp) rather than third-party connectors like Windsor or Supermetrics. Three reasons:

  1. Account safety — Meta has started cracking down on unauthorized MCP usage and blocking ad accounts. Google may follow. The official server is the only one Google themselves built and approved.
  2. No ongoing cost — Windsor and Supermetrics charge monthly fees. The official MCP is free forever.
  3. Full data access — The official server supports GAQL (Google Ads Query Language), giving you access to every report and metric the API offers. Third-party connectors often expose a subset.

Set up the OAuth consent screen

This tells Google what your app is and who can use it. Without this, authentication will fail.

This is the #1 reason installs fail. If you skip or misconfigure this step, you'll get "This app is blocked" later and have to come back here anyway.
a

In Google Cloud Console, go to APIs & Services → OAuth consent screen

b

Choose External as the user type and click Create

"Internal" only works for Google Workspace orgs. External works for everyone.

c

Fill in the required fields:

  • App name: anything (e.g., "Google Ads MCP")
  • User support email: your email
  • Developer contact email: your email

Skip optional fields. Click Save and Continue.

d

On the Scopes page, just click Save and Continue (no changes needed).

e

On the Test users page, click Add Users.

Add the email address of the Google account that owns or has access to the Google Ads account.

If the Google Cloud project owner and the Google Ads account owner are different emails, add both.

f

Click Save and Continue, then Back to Dashboard.

4

Create OAuth credentials

These credentials let your machine authenticate with Google on your behalf.

a

In Google Cloud Console: APIs & Services → Credentials

b

Click + Create Credentials at the top → choose OAuth client ID

c

Application type: Desktop app

Name it anything (e.g., "Google Ads MCP Desktop"). Click Create.

d

A popup appears with your Client ID and Client Secret. Click Download JSON.

e

Save the downloaded JSON file somewhere you'll remember.

Suggested location:

Windows: C:\Users\YOUR_USERNAME\.credentials\google-ads-oauth.json
Mac:     ~/.credentials/google-ads-oauth.json
If you already closed the popup and didn't download the JSON, you can recreate it manually. Go to Credentials, click on the OAuth client you just created, and copy the Client ID and Client Secret. Then create a file with this structure: 
{
  "installed": {
    "client_id": "YOUR_CLIENT_ID",
    "client_secret": "YOUR_CLIENT_SECRET",
    "redirect_uris": ["http://localhost"],
    "auth_uri": "https://accounts.google.com/o/oauth2/auth",
    "token_uri": "https://oauth2.googleapis.com/token"
  }
}

Get a Google Ads Developer Token

The Developer Token is what grants API access to Google Ads data. It comes from your MCC (Manager) account.

b

c

d

Once approved, copy the Developer Token from the API Center page.

Also write down your MCC Customer ID. It's the 10-digit number in the top-right of the MCC dashboard (format: XXX-XXX-XXXX). You'll need it without dashes later (e.g., 1234567890).

Authenticate your machine

This step logs your machine into Google so it can access the Ads API on your behalf. It opens a browser window for you to sign in.

Open a terminal and run this command. Replace the path with where you saved the OAuth JSON from Phase 4.

▶ Windows (PowerShell) 
gcloud auth application-default login `
  --client-id-file="C:\Users\YOUR_USERNAME\.credentials\google-ads-oauth.json" `
  --scopes="https://www.googleapis.com/auth/adwords,https://www.googleapis.com/auth/cloud-platform"
PowerShell gotcha: The quotes around the --scopes value are required. Without them, PowerShell treats the comma as a separator and the command fails silently.
▶ Mac / Linux 
gcloud auth application-default login \
  --client-id-file=~/.credentials/google-ads-oauth.json \
  --scopes=https://www.googleapis.com/auth/adwords,https://www.googleapis.com/auth/cloud-platform

A browser window will open. Sign in with the Google account that has access to your Google Ads account. Click "Allow" when prompted.

When it completes, the terminal will print a path to the saved credentials file. It looks like this:

Default credential locations:
Windows: C:\Users\YOUR_USERNAME\AppData\Roaming\gcloud\application_default_credentials.json
Mac/Linux: ~/.config/gcloud/application_default_credentials.json
If you see "This app is blocked": Go back to Phase 3 and make sure you added your email as a Test User on the OAuth consent screen.

Install the Google Ads MCP server

The MCP server is the bridge between Claude and Google Ads. It translates Claude's questions into Google Ads API calls.

▶ Windows

Run these two commands one after the other:

python -m venv C:\Users\%USERNAME%\.venvs\google-ads-mcp

Then:

C:\Users\%USERNAME%\.venvs\google-ads-mcp\Scripts\pip install git+https://github.com/googleads/google-ads-mcp.git
Why not pipx? There's a known bug with pipx run on Windows (pip 25.x). The approach above works reliably.
▶ Mac / Linux

Install pipx if you don't have it:

brew install pipx && pipx ensurepath

pipx will automatically download and run the MCP server when Claude needs it (configured in the next step). No separate install needed.

MCP server installed (no errors in terminal)

The public release of google-ads-mcp has a bug that causes every tool call to hang forever when launched as an MCP stdio subprocess on Windows. The root cause is google.auth.default() deadlocking inside the stdio context. Without this patch, your install will appear to "work" but every query will freeze indefinitely.

Configure Claude Code

Tell Claude Code where to find the MCP server and how to authenticate with Google Ads.

Open or create the file ~/.claude/.mcp.json and add the Google Ads MCP entry.

If the file already has other entries: Add the "google-ads-mcp" block inside the existing "mcpServers" object. Don't replace the whole file — just add alongside existing servers.
▶ Windows

File location: C:\Users\YOUR_USERNAME\.claude\.mcp.json

"google-ads-mcp": {
  "command": "C:\\Users\\YOUR_USERNAME\\.venvs\\google-ads-mcp\\Scripts\\python.exe",
  "args": ["-c", "from ads_mcp.server import run_server; run_server()"],
  "env": {
    "GOOGLE_APPLICATION_CREDENTIALS": "C:\\Users\\YOUR_USERNAME\\AppData\\Roaming\\gcloud\\application_default_credentials.json",
    "GOOGLE_PROJECT_ID": "YOUR_GCP_PROJECT_ID",
    "GOOGLE_ADS_DEVELOPER_TOKEN": "YOUR_DEV_TOKEN",
    "GOOGLE_ADS_LOGIN_CUSTOMER_ID": "YOUR_MCC_ID_NO_DASHES"
  }
}
▶ Mac / Linux

File location: ~/.claude/.mcp.json

"google-ads-mcp": {
  "command": "pipx",
  "args": ["run", "--spec", "git+https://github.com/googleads/google-ads-mcp.git", "google-ads-mcp"],
  "env": {
    "GOOGLE_APPLICATION_CREDENTIALS": "/Users/YOUR_USERNAME/.config/gcloud/application_default_credentials.json",
    "GOOGLE_PROJECT_ID": "YOUR_GCP_PROJECT_ID",
    "GOOGLE_ADS_DEVELOPER_TOKEN": "YOUR_DEV_TOKEN",
    "GOOGLE_ADS_LOGIN_CUSTOMER_ID": "YOUR_MCC_ID_NO_DASHES"
  }
}

Replace the placeholders

Placeholder Where to find it
YOUR_USERNAME Your Windows/Mac username (the one in your home folder path)
YOUR_GCP_PROJECT_ID From Phase 2 — the Project ID on your Google Cloud dashboard
YOUR_DEV_TOKEN From Phase 5 — the Developer Token from Admin → API Center
YOUR_MCC_ID_NO_DASHES From Phase 5 — your MCC Customer ID, 10 digits, no dashes (e.g., 1234567890)
Windows paths need double backslashes (\\) in JSON files. Single backslashes won't work.

Restart Claude Code after saving the file.

Register each skill with Claude Code

Claude Code looks for skills at ~/.claude/skills/<skill-name>/SKILL.md. The downloaded skills are nested one folder deeper, so we need to create shortcuts that point to them.

▶ Mac / Linux / Git Bash on Windows 
cd ~/.claude/skills
for dir in google-ads-skills/skills/google-ads-*; do

cd $env:USERPROFILE\.claude\skills
Get-ChildItem google-ads-skills\skills\google-ads-* -Directory | ForEach-Object {
  New-Item -ItemType SymbolicLink -Path $_.Name -Target $_.FullName

Troubleshooting

▶ "This app is blocked" during authentication
Your Google account isn't listed as a test user. Go back to Phase 3: GCP → APIs & Services → OAuth consent screen → Test users → add your email. If the GCP owner and Google Ads owner are different emails, add both.
▶ "Permission denied" or empty results when querying
Check that GOOGLE_ADS_LOGIN_CUSTOMER_ID in your config is the MCC Customer ID (10 digits, no dashes). This field is required when accessing accounts through a Manager account.
▶ Tool calls hang forever (most common Windows issue)

Symptoms: the MCP server appears to load fine, tools/list returns instantly, but the moment you ask Claude a real question (e.g., "list my campaigns"), nothing happens. The query just hangs. No error, no timeout, no progress.

Cause: google.auth.default() deadlocks inside the MCP stdio subprocess. This is a bug in the public release of google-ads-mcp on Windows.

Fix: apply the stdio-hang patch from Phase 7. Re-read the red callout in Phase 7 and run:

C:\Users\%USERNAME%\.venvs\google-ads-mcp\Scripts\python.exe C:\tools\gads-installer\apply-stdio-hang-patch.py C:\Users\%USERNAME%\.venvs\google-ads-mcp

Then run verify-mcp-health.py to confirm. Restart Claude Code after patching — the old MCP server process is cached in memory and needs to be killed.

Re-run after every reinstall of google-ads-mcp — pip overwrites the patched file.

▶ "No module named pip.__main__" on Windows
This is a known pipx bug with pip 25.x on Windows. Don't use pipx. Use the venv approach from Phase 7 (Windows tab) instead.
▶ MCP server not showing up in Claude
  1. Did you restart Claude Code / Claude Desktop after editing the config file?
  2. Check the JSON file for syntax errors — a missing comma or unclosed bracket will break the whole file
  3. On Windows: make sure all file paths use double backslashes (\\), not single
  4. Paste the JSON into a validator like jsonlint.com to check
▶ Developer token still pending after 48 hours
Go back to ads.google.com → Admin → API Center in your MCC account and check the status. Make sure you applied from the Manager (MCC) account, not a regular account. If still stuck, contact Google Ads support.
▶ gcloud auth command fails with scopes error (PowerShell)
PowerShell treats the comma in --scopes as an array separator, which breaks the command. Make sure the entire value is wrapped in double quotes: --scopes="scope1,scope2". See Phase 6 for the exact command.
▶ Skills not loading (Claude gives generic answers, not structured audits)
  1. Check that the skill shortcuts were created: ls ~/.claude/skills/google-ads-* should show 21 entries
  2. Each should contain a SKILL.md file. Check one: ls ~/.claude/skills/google-ads-account-audit/
  3. If missing, re-run Phase 10 step b

Good to know

Read-only

Claude can analyze data but cannot create, pause, or change campaigns. Recommendations only.

On-demand only

Claude responds when you ask. It doesn't monitor the account or send alerts on its own.

Performance Max is limited

PMax campaigns have restricted data in the API — search terms, asset breakdowns, and channel splits are not fully available.

~3 hour data lag

Google Ads data has an inherent delay. Today's data may be incomplete until late afternoon.

API rate limits

Very large accounts with many queries in one session may hit Google's API limits.

Conversion numbers may differ slightly

API conversion data can differ from the Google Ads UI due to attribution model differences. This is a Google-side thing.
